                                   JSON  { steer_deg: float, speed_pct: float }
                                   Watchdog: 500 ms silence → Robot Agent sends e-stop
    Dashboard    → Robot Agent  :  HTTP GET   /stream   (MJPEG multipart)
    Dashboard    → Robot Agent  :  HTTP GET   /status   (JSON telemetry, served from cache)
    Robot Agent  → dxl_commander:  USB Serial binary frames, 115200 baud
                                   Jetson→OpenCM fixed-length packet:
                                     header | seq | timestamp_ms | 8× (mode, target, enable_torque) | payload | CRC
    dxl_commander→ Robot Agent  :  USB Serial binary frames, 115200 baud
                                   OpenCM→Jetson fixed-length packet:
                                     header | seq | timestamp_ms | e-stop_state | IMU (reserved) | 8× (available, mode, pos, speed, temp, voltage) | payload | CRC
    dxl_commander→ AX-12A       :  Dynamixel Protocol 1.0, 1 Mbaud

  8. ARCHITECTURE DECISION RECORDS
================================================================================

  ──────────────────────────────────────────────────────────────────────────
  ADR-001  Use WebSocket for drive command channel
  ──────────────────────────────────────────────────────────────────────────
  Status  : Accepted
  Date    : 2026-05-11

  Context
    Drive commands must be sent from the operator browser to the robot at
    ~50 Hz with minimal latency. The robot does not need to respond to each
    command (fire-and-forget); however the channel must be persistent and
    low-overhead.

  Decision
    Use a persistent WebSocket connection (JSON messages) on FastAPI's
    native WebSocket route.

  Consequences
    + Full-duplex persistent connection — no HTTP handshake overhead per frame.
    + Native browser API (no library needed).
    + FastAPI natively supports WebSocket with async handlers.
    - JSON parsing has marginal overhead vs. binary; acceptable at 50 Hz.
    - Requires connection-loss handling logic on the dashboard.

  ──────────────────────────────────────────────────────────────────────────
  ADR-002  Use MJPEG over HTTP for camera streaming
  ──────────────────────────────────────────────────────────────────────────
  Status  : Accepted
  Date    : 2026-05-11

  Context
    The operator needs a live camera feed rendered in the browser.
    Options evaluated: WebRTC, HLS, WebSocket binary, MJPEG over HTTP.

  Decision
    Serve MJPEG as a multipart HTTP stream from FastAPI's StreamingResponse.
    Consume it in the browser via a plain <img src="..."> tag.

  Consequences
    + Zero browser-side code required (native <img> support).
    + No signalling server or STUN/TURN infrastructure (unlike WebRTC).
    + OpenCV's imencode() produces JPEG frames natively.
    - One-way stream only; no audio.
    - Higher bandwidth than H.264 (WebRTC). Acceptable for LAN/VPN.
    - Latency ~100–300 ms. Sufficient for teleoperation; revisit for v2.

  ──────────────────────────────────────────────────────────────────────────
  ADR-003  Use Tailscale for remote connectivity
  ──────────────────────────────────────────────────────────────────────────
  Status  : Accepted
  Date    : 2026-05-11

  Context
    The robot operates behind NAT (home router, 4G modem, etc.). The operator
    must be able to reach the robot over the internet without a static IP or
    port-forwarding. Options: self-hosted relay (nginx+TURN), ngrok, Tailscale.

  Decision
    Use Tailscale on both the Jetson and the operator machine. The robot
    gets a stable 100.x.x.x VPN IP. The operator connects to that IP directly.

  Consequences
    + No server infrastructure to build or maintain.
    + WireGuard encryption by default.
    + NAT traversal handled automatically.
    + Free tier supports this use case.
    - Requires Tailscale account (third-party dependency).
    - Both machines must have Tailscale installed and authenticated.
    - Not fully self-hosted (can migrate to Headscale in v2 if needed).

  ──────────────────────────────────────────────────────────────────────────
  ADR-004  Use dxl_commander firmware (text serial) over host-side SDK
  ──────────────────────────────────────────────────────────────────────────
  Status  : Accepted
  Date    : 2026-05-11

  Context
    Dynamixel servos can be controlled from the host (Jetson) using the
    official DynamixelSDK (Python), or controlled via a microcontroller
    running Dynamixel2Arduino, with the host sending higher-level commands.

  Decision
    All Dynamixel complexity lives in the dxl_commander firmware on the
    OpenCM9.04. The host (Jetson) sends newline-terminated text commands
    over USB Serial and receives structured text responses. No DynamixelSDK
    on the host.

  Consequences
    + Host code is decoupled from Dynamixel protocol details.
    + Firmware already built and tested.
    + Text protocol is language-agnostic and debuggable with any serial terminal.
    + OpenCM9.04 handles half-duplex bus timing natively.
    - Extra USB-Serial hop adds ~1–2 ms latency per command.
    - Firmware update required to add new servo operations.

  ──────────────────────────────────────────────────────────────────────────
  ADR-005  Use FastAPI + uvicorn for the Robot Agent
  ──────────────────────────────────────────────────────────────────────────
  Status  : Accepted
  Date    : 2026-05-11

  Context
    The Robot Agent needs to serve: a WebSocket endpoint, an HTTP streaming
    response (MJPEG), and a JSON REST endpoint — all concurrently.
    Options evaluated: Flask, aiohttp, FastAPI, ROS2.

  Decision
    Use FastAPI with uvicorn. All three endpoint types are natively supported
    in a single async process.

  Consequences
    + Async-native: WebSocket and streaming run concurrently without threads.
    + WebSocket support built-in (no extension needed, unlike Flask).
    + StreamingResponse handles MJPEG cleanly.
    + Minimal boilerplate.
    - ROS2 would provide topic/service infrastructure useful in v2 (SLAM,
      odometry). This is a deliberate deferral — FastAPI is simpler for v1
      and can coexist with ROS2 in a future version.

  ──────────────────────────────────────────────────────────────────────────
  ADR-007  Use binary fixed-length frames for Jetson ↔ OpenCM serial link
  ──────────────────────────────────────────────────────────────────────────
  Status  : Accepted
  Date    : 2026-05-16

  Context
    The original text-based serial protocol (SETPOS, SETSPEED, GETPOS …)
    sends one command per servo per direction per tick. At 50 Hz with 8
    servos, this produces up to 400 serial round-trips per second at
    115200 baud, saturating the link and adding per-command ACK overhead.
    The data_structure.txt already described a binary packet design that
    bundles all 8 servo targets in one frame.

  Decision
    Replace the text protocol with a fixed-length binary frame pair:
      Jetson → OpenCM : header | seq | timestamp_ms | 8× (mode, target,
                         enable_torque) | 64-byte payload | CRC
      OpenCM → Jetson : header | seq | timestamp_ms | e-stop_state |
                         IMU fields (reserved) | 8× (available, mode,
                         current_pos, current_speed, temp, voltage) |
                         64-byte payload | CRC
    The Robot Agent sends one frame per control tick (~50 Hz). The OpenCM
    replies with one state frame per received command. The Serial Driver
    runs recv_loop() as a background asyncio task and caches the latest
    state, making /status instant.

  Consequences
    + Bus utilisation drops from ~400 round-trips/s to ~50 writes/s.
    + All servo targets dispatched atomically in one write.
    + State cache eliminates serial blocking from /status handler.
    + CRC catches bit errors on the USB-serial link.
    - Firmware must be updated to parse/emit binary frames (replaces
      text command parser in dxl_commander).
    - Binary protocol is less debuggable without tooling (add a debug
      mode that echoes frames as hex if needed).
    - Requires agreed frame layout documented in data_structure.txt.

  ──────────────────────────────────────────────────────────────────────────
  ADR-008  WebSocket watchdog and dual-layer e-stop
  ──────────────────────────────────────────────────────────────────────────
  Status  : Accepted
  Date    : 2026-05-16

  Context
    If the operator's WebSocket connection drops (VPN hiccup, browser
    crash, network loss), the robot holds its last commanded state:
    wheels spinning at speed, steering held at angle. There is no
    mechanism in v1 to detect this and bring the robot to a safe stop.

  Decision
    Implement a two-layer e-stop:

    Layer 1 — Robot Agent (software watchdog):
      The E-Stop Handler component monitors the timestamp of the last
      received WebSocket message. If >500 ms elapse with no message,
      it sends a zero-speed binary frame via the Serial Driver and sets
      the e-stop_state flag in the outbound packet.

    Layer 2 — dxl_commander (firmware watchdog):
      The firmware tracks time since the last valid received frame. If
      >500 ms elapse with no frame, it independently sets all drive
      speeds to zero and all steering to neutral. This layer fires even
      if the Jetson crashes or reboots.

    OpenCM is powered directly from the 12V rail via its onboard power
    switch (the system master switch). This is independent of Jetson power,
    so Layer 2 fires correctly even on a Jetson crash or reboot. The
    Jetson→OpenCM USB cable carries serial data only — no power.

  Consequences
    + Robot stops safely on any network or software fault.
    + Layer 2 protects against Jetson-level crashes.
    + e-stop_state flag in state frame allows dashboard to show alert.
    - 500 ms threshold must be tuned: too short → false trips on jitter;
      too long → robot travels further before stopping.
    - Firmware update required to implement Layer 2 watchdog.
