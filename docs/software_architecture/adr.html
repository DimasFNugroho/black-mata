ADR-001 Use WebSocket for the drive command channel
Accepted 2026-05-11
▶
Context

Drive commands must be sent from the operator browser to the robot at approximately 50 Hz with minimal round-trip latency. The channel must be persistent and low-overhead since a new connection per frame would be impractical. Options evaluated: HTTP polling, Server-Sent Events, WebSocket, MQTT over WebSocket.

Decision

Use a persistent WebSocket connection on a FastAPI /ws route. Each message carries a JSON object: {steer_deg: float, speed_pct: float}.

Consequences
  • Full-duplex persistent connection — no HTTP handshake overhead per control frame.
  • Native browser WebSocket API — no client library required.
  • FastAPI supports WebSocket natively with async handlers alongside HTTP routes.
  • JSON serialization has marginal overhead over a binary protocol; acceptable at 50 Hz.
  • Requires reconnection logic on the dashboard side for dropped connections.
ADR-002 Use MJPEG over HTTP for camera streaming
Accepted 2026-05-11
▶
Context

The operator needs a live camera feed rendered in the browser. Four options were evaluated: WebRTC (peer-to-peer, H.264), HLS (segmented streaming), WebSocket binary frames, and MJPEG over HTTP.

Decision

Serve MJPEG as a multipart/x-mixed-replace HTTP stream via FastAPI's StreamingResponse. The Dashboard consumes it with a plain <img src="/stream"> tag — no JavaScript involved.

Consequences
  • Zero browser-side code — the <img> tag handles the stream natively.
  • No STUN, TURN, or signalling infrastructure needed (unlike WebRTC).
  • OpenCV imencode() produces JPEG frames natively — no additional pipeline needed.
  • Higher bandwidth than H.264. Acceptable over LAN and VPN for v1; revisit in v2.
  • Latency approximately 100–300 ms. Sufficient for teleoperation.
  • No audio channel over MJPEG.
ADR-003 Use Tailscale for remote connectivity
Accepted 2026-05-11
▶
Context

The robot operates behind NAT — home router, 4G modem, university network — and does not have a static IP. The operator must reach the robot from anywhere without port forwarding. Options evaluated: self-hosted reverse proxy with TURN (nginx), ngrok tunnels, Tailscale, Cloudflare Tunnel.

Decision

Install Tailscale on both the Jetson and the operator machine. The robot receives a stable 100.x.x.x VPN address. The operator connects directly to that address. No relay server code is written or maintained.

Consequences
  • Zero infrastructure to build or maintain.
  • WireGuard encryption by default — end-to-end between operator and robot.
  • NAT traversal and reconnection handled automatically.
  • Free tier supports this use case indefinitely.
  • Third-party dependency — requires a Tailscale account.
  • Both machines must have Tailscale installed and authenticated before use.
  • Not fully self-hosted. Can migrate to Headscale (self-hosted Tailscale control plane) in v2 if required.
ADR-004 Use dxl_commander firmware instead of host-side DynamixelSDK
Accepted 2026-05-11
▶
Context

AX-12A servos can be controlled from the Jetson using the official Python DynamixelSDK directly over a U2D2 USB adapter, or they can be driven via an OpenCM9.04 running Dynamixel2Arduino, with the Jetson sending higher-level text commands over USB Serial. The OpenCM9.04 hardware is already present in the robot.

Decision

All Dynamixel protocol complexity lives inside the dxl_commander firmware on the OpenCM9.04. The Jetson sends newline-terminated ASCII commands such as SETPOS 1 512 and receives structured text responses. No DynamixelSDK is installed on the host.

Consequences
  • Host code is fully decoupled from Dynamixel protocol internals and baud timing.
  • The firmware is already built and tested — it is an existing asset.
  • Text protocol is language-agnostic and can be debugged from any serial terminal.
  • OpenCM9.04 handles half-duplex bus direction switching natively and reliably.
  • Extra USB-Serial hop adds approximately 1–2 ms latency per command batch.
  • Adding a new servo operation (e.g. sync write) requires a firmware update cycle.
ADR-005 Use FastAPI + uvicorn for the Robot Agent
Accepted 2026-05-11
▶
Context

The Robot Agent must concurrently serve a WebSocket endpoint, an HTTP streaming response (MJPEG), and a JSON REST endpoint — all from a single process on the Jetson. Options evaluated: Flask, aiohttp, FastAPI, and ROS2 (which includes its own communication layer).

Decision

Use FastAPI with uvicorn. All three endpoint types are natively supported within a single async process.

Consequences
  • Async-native: WebSocket, streaming, and REST handlers run concurrently without threads.
  • WebSocket support is built into FastAPI with no extension needed.
  • StreamingResponse cleanly wraps the MJPEG generator.
  • Minimal boilerplate — fast to iterate and easy to read.
  • ROS2 would provide topic and service infrastructure that will be useful in v2 when odometry, SLAM, and waypoints are added. This is a deliberate deferral for v1 simplicity. FastAPI can coexist with a future ROS2 layer via a bridge node.
ADR-006 Use Vanilla JS for the Operator Dashboard
Accepted 2026-05-11
▶
Context

The dashboard must display a live camera stream, read gamepad and keyboard input, maintain a WebSocket connection for sending drive commands, and show a servo status panel. Options evaluated: React, Vue, Svelte, and Vanilla JS with browser built-ins.

Decision

A single HTML file using only built-in browser APIs: Gamepad API for controller input, WebSocket API for the command channel, Fetch API for polling telemetry, and standard DOM manipulation. No build toolchain.

Consequences
  • No build step, no npm, no bundler — the file opens directly from the filesystem or is served as a static file.
  • Browser Gamepad API natively polls controller state — no library needed.
  • All five v1 dashboard requirements are satisfied by browser built-ins alone.
  • Easy to serve as a static file from the Robot Agent's FastAPI instance.
  • Not suitable if the dashboard grows substantially in v2 (maps, waypoint editor, tuning panels). Revisit with a lightweight framework at that point.
